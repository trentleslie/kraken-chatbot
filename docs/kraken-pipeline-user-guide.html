Biomedical Knowledge-Graph Discovery

KRAKEN Discovery Pipeline

A 10-node LangGraph workflow that turns a biomedical question into a research-grade analysis — resolving entities, classifying them by knowledge-graph connectivity, reasoning over the Kestrel graph at one-hop / multi-hop / subgraph depth, and synthesising a cited report with hypotheses.

1 · Overview

KRAKEN answers biomedical discovery questions against Kestrel, a large biomedical knowledge graph. It runs as a Python (FastAPI) backend plus a React frontend that communicate over WebSocket. Two engines sit behind it: a single-agent Classic mode for direct graph queries, and a Discovery Pipeline mode — the LangGraph workflow this guide documents.

The pipeline is additive and degradation-tolerant: each node enriches a shared state object, and individual API/LLM failures are logged and skipped rather than aborting the run.

2 · Operating modes

Classic mode

A single Claude agent with the Kestrel MCP tools, for direct, conversational graph queries ("what is X associated with?"). Fast, interactive, no multi-node orchestration.

Note: the "advanced" agent tools (guilt_by_association, missing_edge_prediction, …) are agent-side Claude prompts, not Kestrel server endpoints.

Discovery Pipeline mode

The 10-node LangGraph workflow — research-grade analysis with entity triage, parallel well-characterised vs. cold-start branches, pathway enrichment, cross-entity integration, optional temporal classification, synthesis, and literature grounding.

This guide covers Discovery Pipeline mode.

3 · The discovery pipeline

Entry point is intake; the graph ends after literature_grounding. Two decision points shape the flow: Triage routes entities into parallel analysis branches by connectivity, and Integration conditionally inserts the Temporal node for longitudinal studies.

If triage resolves no entities, it routes straight to pathway_enrichment (analysis branches skipped). When both well-characterised and sparse entities exist, direct_kg and cold_start run in the same LangGraph superstep; their findings merge safely via operator.add reducers.

5 · Reasoning depth

The pipeline reasons over the graph at three depths. One-hop is always on; multi-hop and subgraph are flag-gated and default off (the "reasoning-depth demo slice"), so they stay inert until explicitly enabled.

ModeWhereWhat it surfacesStatus
One-hopdirect_kgDirect disease/pathway associations via ranking presets.always on
Multi-hopdirect_kgMechanistic chains for well-characterised entities, e.g. disease → intermediate → target (source=direct_kg_multi_hop).multi_hop_enabled
SubgraphintegrationConnecting structure (nodes + edges) between resolved entities, summarised as a Bridge.subgraph_enabled
Hub guard. Multi-hop and subgraph queries suppress generic "hub" nodes in-query via a degree < hub_threshold constraint (Kestrel accepts degree / degree_percentile node constraints), so deep traversal doesn't resurface high-degree noise as findings.

6 · Two-tier execution

Most analysis nodes follow the same resilience pattern:

Tier 1 — direct API

Structured calls to Kestrel (one_hop_query, multi_hop_query, similar_nodes, …). Fast, deterministic, the common path.

Tier 2 — Claude SDK fallback

When a Tier-1 call fails, the node falls back to a Claude Agent SDK reasoning step with the Kestrel MCP tools. Bounded by per-node semaphores.

Concurrency is deliberately heterogeneous: entity_resolution / triage use a semaphore of 1 (CLI-spawn safety); batch-analysis nodes use 6–8. The demo-slice multi-hop fan-out has its own dedicated semaphore so it stays independent of Tier-2 batching.

Per-node settings live in backend/src/kestrel_backend/graph/pipeline_config.py (accessed via get_pipeline_config()). Key knobs:

Config · fieldDefaultMeaning

DirectKGConfig.multi_hop_max_hops / _limit2 / 10Multi-hop path length and result cap.
DirectKGConfig.multi_hop_semaphore6Concurrent multi-hop calls (independent of Tier-2 batching).
IntegrationConfig.subgraph_enabledFalseEnable connecting-subgraph detection between entities.
IntegrationConfig.max_subgraph_nodes5Entities passed as node_ids to a subgraph query.

Demo flags are off by default. To show reasoning depth (e.g. in LangGraph Studio), set multi_hop_enabled and subgraph_enabled to True, then revert after — they gate behaviour whose quality has not yet been formally evaluated.

9 · Inputs & outputs

The pipeline operates on a single shared DiscoveryState (a Pydantic-typed TypedDict). You invoke it with a query string; it returns the final state.

from kestrel_backend.graph.runner import run_discovery

state = await run_discovery("Type 2 diabetes mechanisms and connected drug targets")
print(state["synthesis_report"])   # the narrative report
print(state["hypotheses"])         # list[Hypothesis], each with literature_support
print(state["bridges"])            # list[Bridge] cross-entity connections

Key state fields:

FieldTypeDescription
raw_querystrThe input question (required).
query_typeretrieval · discovery · hybridDetected intent.
direct_findingslist[Finding]Evidence from direct_kg / cold_start / integration (incl. multi-hop chains).
bridgeslist[Bridge]Cross-entity connections (multi-hop + subgraph).
synthesis_reportstrFinal narrative report.
hypotheseslist[Hypothesis]Generated hypotheses, grounded with citations.
model_usageslist[ModelUsageRecord]Per-node token usage (for cost tracking).

Streaming: stream_discovery(query) yields {node_name: node_output} per superstep — this is what LangGraph Studio visualises.

10 · Running it

Local backend + frontend

# Frontend (React, Vite HMR on :5173)
npm run dev

# Backend (FastAPI, WebSocket on :8000)
cd backend && uv sync
uv run uvicorn src.kestrel_backend.main:app --reload --host 127.0.0.1 --port 8000

LangGraph Studio (visual debugger)

cd backend && uv run langgraph dev    # open the printed Studio URL
# graph entry: kestrel_backend.graph.builder:build_discovery_graph (kraken_discovery)

Tests

cd backend
uv run python -m pytest tests/ -m "not integration"
Use uv run python -m pytest — not uv run pytest. On paths containing spaces, the pytest launcher's shebang breaks and falls back to the system Python (which lacks the project's dependencies).

Required environment (backend/.env)

KESTREL_API_KEY (graph data access), ANTHROPIC_API_KEY + Claude SDK OAuth (claude login), and the literature keys S2_API_KEY / EXA_API_KEY (OpenAlex & PubMed need none). Set CLERK_AUTH_ENABLED=false to disable auth locally.
