# feat: research-register prose for discovery synthesis + performance reports

## Overview

Make the discovery pipeline emit its user-facing prose in the "spiral nucleic-acids research register" — the academic writing-style guide currently documented only in the `publish-wiki` skill and applied by hand at wiki-publish time. After this change the pipeline produces reports already in that register, so no manual restyling is needed downstream.

Two surfaces are in scope:
1. **Discovery synthesis report** — LLM-generated by the synthesis node; the register is injected into `SYNTHESIS_PROMPT`.
2. **Performance report** — its *prose* (the wall-clock caveat and a short derived summary) in `graph/performance_report.py:render_markdown`; tables and labels stay literal.

The register lives as **one canonical backend constant** that both surfaces import (and that the classic chatbot can adopt later). The classic chatbot mode is **deferred** to a follow-up.

## Problem Frame

The research register is real institutional style (used for Phenome wiki deliverables) but exists only as prose inside the `publish-wiki` skill. Pipeline outputs are written in an ad-hoc "clear, scientific" voice and only get the register when a human (or this agent) rewrites them at publish time — which is lossy, manual, and non-reproducible. Baking the register into the generators makes every run's output publish-ready and consistent.

Builds on the just-merged per-node performance report (PR #84) and the synthesis-overflow caps (PR #85). The user clarified the target is the pipeline's *actual outputs* (synthesis + performance report), classic chatbot deferred.

## Requirements Trace

- **R1.** The discovery synthesis report is generated in the research register (register injected into `SYNTHESIS_PROMPT`), without breaking its existing section structure or evidence-tag rules.
- **R2.** The performance report's prose (caveat + a short summary) reads in the register; tables, labels, and the `run_id`/query-handling are unchanged.
- **R3.** The register is defined **once** as a canonical backend constant. Synthesis imports it at runtime (injected into `SYNTHESIS_PROMPT`); the performance-report prose is hand-authored to match it (the constant is the canonical reference text, not embedded in render output); the classic agent will import it when un-deferred. No duplicated register prose, no string drift. The constant's docstring cites the `publish-wiki` source for cross-repo sync.
- **R4.** The register addition does not regress the synthesis context budget. It enlarges `SYNTHESIS_PROMPT` (the *system prompt*), which shares the model's ~200K-token window with the assembled context and the output. PR #85's `SynthesisConfig.max_context_chars` (pipeline_config.py) caps the *context* — a char proxy for tokens — and was sized to leave headroom for "system prompt + output." The register's added system-prompt size must stay within that headroom (reduce `max_context_chars` by the register's char-equivalent if the margin is tight), and synthesis must not start tripping the empty-output fallback marker.
- **R5.** Classic chatbot mode (`agent.py` `SYSTEM_PROMPT`) is explicitly deferred but the constant is structured so it can adopt the register with a one-line append later.

## Scope Boundaries

- Not restyling structured/JSON-producing nodes (`pathway_enrichment`, `cold_start`, `integration`) — they emit data, not user prose (verified: only synthesis + classic agent produce narrative).
- Not restyling tables, labels, code blocks, or CLI-like fragments in any output (per the publish-wiki register's own rule — prose only).
- Not changing the synthesis report's section structure, evidence tags, headline logic, or the performance report's schema/metrics/`run_id` handling.
- Not building a runtime sync mechanism to the publish-wiki skill or a "spiral app" — the kraken constant is the runtime source of truth; sync to the publish-wiki text is a documented, manual concern.

### Deferred to Separate Tasks

- **Classic chatbot register** (`agent.py` `SYSTEM_PROMPT`, line 119): a follow-up; likely a one-line append of the shared constant, possibly with a "keep short conversational replies natural" scoping sentence. Deferred per the user's scope choice.

## Key Technical Decisions

- **One canonical constant in a dedicated module** `backend/src/kestrel_backend/writing_style.py` (e.g. `RESEARCH_REGISTER`), imported by `synthesis.py` and `performance_report.py` (and later `agent.py`). Avoids string drift (R3). Module docstring cites the `publish-wiki` "spiral nucleic-acids research register" as the upstream source to keep in sync.
- **Condensed, faithful register text**, not the full publish-wiki prose — to protect the tight synthesis budget (R4). Capture the load-bearing rules (subject-first sentences; semicolons/colons over em-dashes; precise domain vocabulary; exact numbers; measured authoritative tone; limitations stated explicitly) in a compact block (target ≤ ~1200 chars / ~340 tokens; the Unit 1 size-ceiling test enforces this).
- **The register block must declare its own subordination**: include a sentence inside the constant stating these style rules govern *prose voice only* and are subordinate to all structural directives (mandatory sections, the `[KG Evidence]`/`[Literature]`/`[Model Knowledge]`/`[Inferred]` tags, tier requirements, tables) — apply the register only to narrative prose within those constraints. This ships in the constant, not just as implementation guidance, so the synthesis LLM sees it.
- **Pronoun adaptation for automated reports**: the publish-wiki register's "use *we* for author actions" was written for human-authored papers; in an LLM-generated discovery report "we analyzed…" is ambiguous (pipeline vs. researcher). The kraken adaptation should prefer measured impersonal/observational voice (e.g. "the module encodes…", "the data indicate…") rather than first-person "we," and say so explicitly in the constant.
- **Synthesis injection by appending** the register as a stable, clearly-headed block in `SYNTHESIS_PROMPT` (e.g. a `## Writing register` section) — positioned so it does not contradict the existing section/evidence-tag rules, and placed for cache-friendliness (stable system-prompt content).
- **Synthesis budget reconciliation** (against the *real* PR #85 mechanism): the budget control is `SynthesisConfig.max_context_chars` (pipeline_config.py:~355) — a char cap on the assembled **context**, a proxy for the token window, sized to leave headroom for the system prompt + output. The register grows the *system prompt*, not the context, so it eats that headroom. Verify the headroom still covers the enlarged prompt; reduce `max_context_chars` by the register's char-equivalent if tight. There is **no** "system-prompt reserve" field to adjust — do not look for one.
- **Branch prerequisite**: this feature must branch off `dev` **after PR #85 is merged** (origin/dev `0a63694`). PR #85 adds `SynthesisConfig`/`max_context_chars`; a branch off the older #84-era dev will not have it, and Unit 2's reconciliation would have nothing to reason about.
- **Performance report**: restyle only the prose in `render_markdown` (rewrite the caveat in-register; add a compact register-voice summary sentence derived from `totals`/`headline`, e.g. naming the dominant latency and cost contributors). Tables, labels, and `run_id`/query handling unchanged (R2).
- **Register applies to prose only** everywhere — tables/labels/code stay literal (per the register's own rule).

## Implementation Units

- [ ] **Unit 1: Canonical research-register constant**

**Goal:** A single backend constant holding the condensed research register, importable by all prose surfaces.

**Requirements:** R3

**Dependencies:** None

**Files:**
- Create: `backend/src/kestrel_backend/writing_style.py` (`RESEARCH_REGISTER` string constant + module docstring citing the publish-wiki source)
- Test: `backend/tests/test_writing_style.py`

**Approach:**
- Condense the publish-wiki "spiral nucleic-acids research register" into a compact, prompt-ready block capturing the load-bearing rules (subject-first sentences with active verbs; semicolons/colons over em-dashes; precise domain nominals; exact numbers; measured authoritative tone; limitations stated explicitly). Target ≤ ~1200 chars.
- Include two adaptations for the automated-report context (see Key Technical Decisions): (1) a **subordination clause** — these style rules govern prose voice only and are subordinate to the structural/section/evidence-tag/table rules; (2) **impersonal voice** — prefer observational phrasing ("the module encodes…", "the data indicate…") over first-person "we," which is ambiguous in an LLM-generated report.
- Plain module-level `str` constant; no templating. Docstring: "Kept in sync with the publish-wiki 'spiral nucleic-acids research register'; update both together."
- Import paths (asymmetric by depth): `synthesis.py` → `from ...writing_style import RESEARCH_REGISTER` (3 levels); a future `agent.py` consumer → `from .writing_style import RESEARCH_REGISTER` (same package). `performance_report.py` does **not** import it (its prose is hand-authored to match — see Unit 3).

**Patterns to follow:** existing module-level prompt constants (`synthesis.py:SYNTHESIS_PROMPT`); leaf module with no internal imports (no cycle risk).

**Test scenarios:**
- Happy path: `RESEARCH_REGISTER` is a non-empty str, importable from `kestrel_backend.writing_style`.
- Edge case (budget guard, R4): assert `len(RESEARCH_REGISTER) <= 1200` chars so future edits can't silently bloat the synthesis prompt.
- Happy path: contains the core directives — assert substrings for the em-dash rule, the subordination clause, and the impersonal-voice instruction — so a careless rewrite that drops them fails.

**Verification:** `synthesis.py` imports the one constant; the perf-report prose is authored against it; no duplicated register prose exists in the codebase.

- [ ] **Unit 2: Inject the register into the synthesis prompt (budget-safe)**

**Goal:** The synthesis report is generated in the research register, without overflowing the context window or breaking the existing report contract.

**Requirements:** R1, R4

**Dependencies:** Unit 1

**Files:**
- Modify: `backend/src/kestrel_backend/graph/nodes/synthesis.py` (append `RESEARCH_REGISTER` as a `## Writing register` block in `SYNTHESIS_PROMPT`; `from ...writing_style import RESEARCH_REGISTER`)
- Modify (if the headroom is tight): `backend/src/kestrel_backend/graph/pipeline_config.py` — reduce `SynthesisConfig.max_context_chars` by the register's char-equivalent so the context cap still leaves room for the enlarged system prompt + output
- Test: `backend/tests/test_synthesis_register.py` (plus existing synthesis tests in `tests/test_langgraph_prototype.py` must still pass)

**Approach:**
- Append the register as a clearly-headed, stable block after the existing structure/evidence-tag rules. The subordination clause lives *in the constant* (Unit 1), so the block self-declares it governs prose voice only and defers to sections/tags/tables.
- Budget reconciliation against the **real** mechanism: `SynthesisConfig.max_context_chars` (pipeline_config.py) caps the assembled *context* as a char proxy for the token window, sized to leave headroom for system prompt + output. The register grows the *system prompt*, so confirm that headroom still holds; if tight, lower `max_context_chars` by ~ the register's char size. There is no separate "system-prompt reserve" field.

**Execution note:** Characterization-first — measure `len(SYNTHESIS_PROMPT)` before/after and the current `max_context_chars` headroom margin, so the reconciliation is provably correct; save a before/after sample synthesis run to judge register adoption (rerun to rule out SDK-handshake noise).

**Patterns to follow:** the `ClaudeAgentOptions(system_prompt=..., allowed_tools=[], max_turns=1)` shape; do not change `allowed_tools`/`mcp_servers`.

**Test scenarios:**
- Happy path: `SYNTHESIS_PROMPT` contains the `RESEARCH_REGISTER` text (substring assertion) and still contains its existing section + evidence-tag directives (no accidental truncation of the base prompt).
- Edge case (budget): assert the enlarged `SYNTHESIS_PROMPT` + `SynthesisConfig.max_context_chars` still fit the window with output headroom (chars→tokens arithmetic against the real `max_context_chars`). There is no "reserve constant" to assert against — this arithmetic is the gate.
- Error path (no silent degradation): the empty-output fallback marker path is unchanged — a forced-empty LLM response still emits the fallback marker (guards against the register nudging toward empty/short output going undetected). Reuse `tests/test_synthesis_fallback_marker.py` patterns.
- Integration (manual/eval, noted not unit): a sample discovery run's synthesis report visibly adopts the register (subject-first sentences, impersonal voice, no em-dashes) — validated by human review of an auto-saved before/after sample, not a unit assertion.

**Verification:** synthesis builds the enlarged prompt, the `max_context_chars` arithmetic still leaves output headroom, existing synthesis tests pass, and a sample run reads in-register without tripping the fallback.

- [ ] **Unit 3: Apply the register to the performance-report prose**

**Goal:** The performance report's prose reads in the register; tables/labels/`run_id` handling unchanged.

**Requirements:** R2

**Approach:**

This unit is a deliberately light touch — tighten the two caveats, no more. **Do not add a "summary sentence"**: the dominant-latency/cost facts already appear as the first table rows, so a summary would duplicate the table (scope-guardian + adversarial flagged this). If the caveats already fully satisfy the register, this unit legitimately reduces to a near no-op — record that rather than inventing prose.
- Leave the GFM tables, column labels, header bullets, `run_id`, and query handling exactly as-is (register applies to prose only).

**Patterns to follow:** existing `render_markdown` structure; the two `wall_clock_source` conditional caveats already present.

**Test scenarios:**

- Happy path (R2 preserved): markdown still shows `run_id` and never the raw query (carry forward the existing R2 query-handling assertion).

Keep em-dash-absence as a secondary guard.
- Edge case: with `wall_clock_source == "summed_estimate"`, the restyled caveat still communicates the summed-estimate meaning.

**Verification:** both caveats read in-register, the rewrite is asserted affirmatively, tables/labels/`run_id` are structurally unchanged, and all existing performance-report tests pass.

## System-Wide Impact

- **Interaction graph:** Two surfaces touched (synthesis prompt; perf-report renderer). The shared constant is imported at runtime only by synthesis; the perf-report prose is authored to match it. No graph topology, node wiring, or state changes.
- **Error propagation:** Synthesis prompt growth interacts with PR #85's `SynthesisConfig.max_context_chars` — the only real cross-cutting risk; covered by R4 and Unit 2's budget reconciliation. No new failure surfaces in the renderer (pure prose edits).
- **API surface parity:** The classic chatbot (`agent.py`) is the parallel prose surface intentionally left unchanged (deferred); the shared constant makes adopting it later a one-line change. `system_prompt_hash` for the classic conversation is unaffected now (unchanged); it will change when the deferred unit lands (intended audit signal).
- **Integration coverage:** Synthesis prose quality is LLM-driven and validated by a sample run + human review, not unit tests — called out explicitly so it isn't mistaken for unit-tested.
- **Unchanged invariants:** synthesis section structure + evidence tags; performance-report schema, metrics, `run_id`/query handling, and tables; all JSON-producing nodes; classic chatbot behavior.
